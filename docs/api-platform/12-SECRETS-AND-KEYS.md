# Secrets and Keys

## An Explicit, Honest Gap First

**Fact, surfaced directly:** the Technology Baseline's 21 ratified
Engines contain **no dedicated secrets-management Engine** — no
HashiCorp Vault, Infisical, or equivalent has been evaluated or
ratified. This is a genuine gap, not an oversight this document papers
over. Selecting one is a Build-vs-Buy act outside this Board's
authority this phase (same reasoning as `10-API-GATEWAY.md`'s Gateway
gap). Recorded as a new item in `.claude/context/open-questions.md`
(#29).

This document defines the **required secret-management behavior**
that any future Vault-style Engine, or an interim managed-cloud
equivalent, must satisfy — not a specific product.

## Secret Management

**Recommendation.** Every credential the platform holds toward an
external Engine or provider — Keycloak realm secrets, RabbitMQ
credentials, Kill Bill API keys, Fawry/Paymob/WhatsApp BSP
credentials (`03-API-DOMAIN-INVENTORY.md`'s Integration API rows),
service-identity Client Credentials (`08-AUTHENTICATION.md`) — is
stored in a centralized secret store, never embedded in code,
configuration files committed to version control, or environment
variables set by hand outside a managed secrets pipeline. No Module
is permitted to invent its own local secret-storage mechanism.

## Certificates

**Recommendation.** mTLS certificates for service-to-service and
Partner Integration traffic (`11-API-SECURITY.md`) are issued and
tracked centrally, not generated ad hoc per Module. Certificate
issuance authority is a platform-level (not per-Module) function,
consistent with the Single Adoption Point discipline applied to
identity infrastructure generally.

## Rotation

**Recommendation.** Every class of secret — service-identity Client
Credentials, mTLS certificates, JWT signing keys (Keycloak-managed),
Partner API keys, external-provider credentials — has a defined
rotation cadence. **This Board does not assert specific rotation
intervals** (e.g., "90 days") as a Decision or Fact — no
`.claude/context/` artifact or user instruction fixes one, and
inventing a number would violate the No-Guessing Rule. The qualitative
requirement is: rotation must be possible without downtime (no secret
is a manual, service-restart-requiring change), and compromise of any
single credential must be independently revocable without affecting
other credentials of the same class.

## Vault Strategy

No Engine is currently ratified to serve this function (see the gap
above). Until Open Question #29 is resolved, this document establishes
the requirements a future selection must meet: centralized storage,
audited access (every read of a secret is itself an Audit Event,
consistent with Full Auditability), dynamic/short-lived credential
issuance where the counterpart Engine supports it (preferred over
long-lived static secrets), and encryption at rest independent of the
application database's own encryption.

## Key Lifecycle

Standard lifecycle, applied uniformly regardless of which future
Engine implements it:

```
Generation → Secure Distribution → Active Use → Rotation → Revocation → Destruction
```

- **Generation**: never manually chosen/typed; generated by the
  issuing system (Keycloak for tokens/signing keys, the eventual
  secrets Engine for service credentials).
- **Secure Distribution**: a secret is delivered to the Module/service
  that needs it through a channel that never traverses an unencrypted
  path or a human-readable chat/email channel.
- **Active Use**: access is scoped to exactly the Module/service that
  needs it — no shared, platform-wide "master" credential.
- **Rotation**: per the cadence discussion above, without service
  disruption.
- **Revocation**: immediate, on suspected compromise or Break-Glass
  session end, independent of the normal rotation cadence.
- **Destruction**: a retired secret is not merely deactivated but
  removed from the store, with the removal itself an Audit Event.
