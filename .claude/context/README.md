# Context Store — README

## الغرض

هذا المجلد هو **Context Store** مختصر ومنظم للمشروع — وليس الوثيقة المعمارية النهائية
(Software Architecture Document) وليس Project Constitution كاملة.

الهدف منه هو تجميع الحقائق المؤكدة، القرارات المقترحة/المعتمدة، الأسئلة المفتوحة،
والقيود المعروفة أثناء نقاشات المشروع، بحيث يمكن لأي جلسة عمل لاحقة (بشرية أو AI)
أن تلتقط السياق بسرعة دون إعادة شرح كل شيء من الصفر.

## قواعد الاستخدام

1. **يُحدَّث بعد كل قرار مهم** — لا يُترك قديمًا أو غير متزامن مع النقاش الفعلي.
2. **لا يحتوي على كود** — هذا المجلد للسياق والمفاهيم فقط، وليس لتطبيقات أو Snippets إنتاجية.
3. **لا يُستخدم بديلاً عن الوثائق الرسمية** — عند إنشاء SAD (Software Architecture Document)
   لاحقًا، هذه الملفات تُستخدم كمُدخلات (inputs) له، وليست بديلاً عنه.
4. **القرارات النهائية تُربط بملفات ADR** — أي قرار ينتقل من Proposed إلى Accepted يجب أن
   يكون له ADR مقابل (باستخدام skill: `architecture-decision-records`) يُشار إليه من
   `decisions.md`.
5. **كل معلومة يجب أن تحمل إحدى الحالات التالية بوضوح**:
   - `Draft` — فكرة أولية لم تُناقش بعد.
   - `Proposed` — مطروحة للنقاش، لم تُعتمد.
   - `Accepted` — معتمدة رسميًا (يجب أن يقابلها ADR).
   - `Rejected` — نُوقشت ورُفضت (مع توضيح السبب إن أمكن).
   - `Superseded` — كانت معتمدة سابقًا واستُبدلت بقرار لاحق (يُشار إلى القرار الجديد).
6. **لا يجوز تغيير قرار بحالة `Accepted` دون ADR جديد** يوثق سبب التغيير ويحيل القرار
   القديم إلى `Superseded`.

## ملفات المجلد

| الملف | الغرض |
|---|---|
| `vision.md` | الرؤية العامة للمنصة والهدف طويل المدى |
| `architecture-principles.md` | المبادئ المعمارية الأولية (Draft) |
| `decisions.md` | فهرس القرارات (ADR Index) + قرارات أولية Proposed |
| `glossary.md` | المصطلحات والتعريفات الأولية (Draft Definitions) |
| `module-catalog.md` | تصنيفات أولية للـ Modules المحتملة |
| `open-questions.md` | الأسئلة المفتوحة التي تحتاج إجابة قبل أي قرار نهائي |
| `constraints.md` | القيود المؤكدة والمبدئية على النظام |
| `stakeholders.md` | فئات أصحاب المصلحة الأولية |

## طريقة التحديث

- أضف معلومة جديدة في الملف المناسب مع تحديد حالتها (Draft/Proposed/...).
- عند اتخاذ قرار معماري كبير: أنشئ ADR أولاً، ثم اربط `decisions.md` به.
- لا تحذف السياق التاريخي — استخدم `Superseded` بدلاً من الحذف عند تغيّر قرار معتمد.
